<objective>
Extend the existing 3-layer CSS token system (tokens.css → index.css → semantic.css) with (a) missing primitive/semantic tokens identified by the Scout audit, AND (b) a set of Phase 5-enablement tokens that define the visual language for the upcoming UI Application phase (neon palette, glows, card surfaces, elevation, display font, realistic fretboard visuals, app background).

This plan is strictly additive. New tokens for Phase 5 (neon, glow, card, elevation, display, fretboard-visual, bg-app) are DEFINED in tokens.css / semantic.css but NOT consumed anywhere in Phase 4 — Phase 5 is where they get wired into screens. This guarantees Phase 4 renders identically.

Also fix the two CAGED color leaks in generic control CSS (StepperControl, FretRangeControl) and document the dark-only intentional choice + --text-muted contrast concern.

Token naming convention to enforce throughout:
- Primitive layer (tokens.css): literal-scale names — --space-N, --font-weight-{scale}, --icon-{size}, --text-{size}, --radius-{size}, --neon-{hue}, --glow-{hue}-{size}
- Semantic layer (semantic.css): purpose/role names — --interactive-primary, --surface-card, --note-ring, --text-brand, --header-surface, --chip-surface-*
- Do NOT add a light-mode semantic layer (app is intentionally dark-only)
</objective>
