## 1. Context

After Plan G shipped (commits `e917773a` → `c364dc97` + visual baselines `23494cf1`), the live UI surfaced seven follow-on issues, plus one regression introduced earlier in Plan G itself. The user's reports map to:

| Ref | Plan G task | What the user saw |
|---|---|---|
| H1 | G1 (`fit`) | Selects still don't feel right — wants **fixed width sized to longest expected option**, plus an overall layout pass on both tabs because there's "lot of wasted space" |
| H2 | G3 + G4 (snap rename + filter rewrite) | "This feature doesn't work well. At some point it did. Massive regression." — the new strict subset filter rejects too many voicings, and the toggle no longer feels useful |
| H3 | G5 (tab styling) | Underline doesn't visually line up with the horizontal divider rail |
| H4 | G6 (shared StringSetPicker) | The "shared" primitive is inconsistent — chord voicing uses dot-separator like `"1·2·3·4"`, scale 1-string says `"String 1"`, scale 2-strings uses dot-separator like `"1·3"`. The user can't see what's actually shared |
| H5 | NEW BUG | CAGED chord connectors persist after the user selects multiple CAGED shapes (toggle from one shape → "All" → multiple connectors stack) |
| H6 | G11 (borrowed resolver) | Resolver now updates, but borrowed cells in DegreeGrid are still not styled clearly enough — especially when SELECTED, the cell reads as muted/dashed AND selected at once, which is confusing |

H1's "thorough layout review" is the load-bearing piece — it's where wasted space lives and where most of the other complaints have their roots.

## 2. Investigation findings

### H1 — Select width + tab layout waste

**LabeledSelect today:**
- Base trigger: `width: 100%`.
- With `fit`: `width: auto`, `max-width: 100%`, `min-width: 5rem`, `align-self: flex-start` (Plan G1). Sizes to current option content.
- Dropdown content panel: `min-width: var(--radix-select-trigger-width)` (so menu = trigger width, never wider).
- Result: when an option's text is long, the menu cuts off without ellipsis or horizontal overflow.

**Inspector PropGrid today:**
- `.propGrid` is `grid-template-columns: repeat(auto-fit, minmax(min(100%, var(--prop-min-width, 11rem)), 1fr))`.
- Default `--prop-min-width: 11rem` forces every cell to be at least 11rem. For short-content cells (Voicing "Close", Lens "Tones/Lead", Strings "1·2·3·4"), this leaves huge negative space inside each cell.
- The Voicing/Lens/Strings/Lock row uses `<PropGrid columns={7}>` with span 3+2+2+2 = **9** spans into a 7-track grid. The extra spans wrap, creating ambiguous flow.

**SongControls today:**
- Top row: Root (span 3) + Scale (span 3) = full row, balanced.
- Meter row: Beats (1) + Tempo (1) + Length (1) + Preset (3) = 6. Preset's span={3} is overkill given `fit`.
- Editor row (post G11c): editor pane span={6}. Inside, the DegreeGrid + 2-col Duration/Quality + hint stack vertically with consistent gap.

### H2 — Lock-to-scale regression (G3 rename + G4 filter)

**Current filter (post-G4, commit `c363d6c7`):**

```ts
// closeCandidatesAtom
if (get(chordSnapToScaleAtom)) {
  const patternKeys = get(activeScalePatternPositionsAtom);
  if (patternKeys.size > 0) {
    windowed = all.filter((v) =>
      v.positionKeys.every((key) => {
        const fretStr = key.split("-")[1] ?? "0";
        return fretStr === "0" || patternKeys.has(key);
      }),
    );
  }
}
```

**The bug:** `every` is brutally strict. A close voicing with 4 notes where 3 are on the pattern and 1 sits one fret outside gets rejected entirely. The result-set is much smaller than what Plan F's fret-bound filter (Plan F's `min/max within [lo, hi]`) used to return. The user remembers a version that "worked" — that was the fret-bound version, which kept practically-useful voicings whose centroids matched the scale region even if a single note sat just outside.

**The right semantic** (user intent extracted from the complaints): "Restrict to voicings the player can grab without leaving the active hand position." That's better modeled as **all fretted notes must lie inside the pattern's fret window** (lo/hi from `activeScaleWindowAtom`), NOT "every note must coincide exactly with a scale-tone position-key." Voicings can include chord tones that aren't scale tones — that's normal harmony, and rejecting them is wrong.

### H3 — Tab underline misalignment

```css
.tabHeader {
  padding-bottom: var(--space-2, 0.5rem);   /* 0.5rem gap below tab row */
  border-bottom: 1px solid var(--faceplate-divider);  /* divider after the gap */
}
.tab {
  border-bottom: 2px solid transparent;     /* sits at tab's bottom edge */
}
.tab[data-state="active"] {
  border-bottom-color: var(--faceplate-accent);
}
```

The active tab's 2px accent border sits at the tab's bottom edge. The `.tabHeader`'s divider sits 0.5rem (`var(--space-2)`) below that. So the accent underline floats above the rail — they don't share a baseline.

**Fix:** Move the gap so the tab's underline sits ON the divider:
- Option A — Negative margin: add `margin-bottom: -1px` to `.tab` so its 2px border overlaps the rail.
- Option B — Eliminate the gap: set `.tabHeader { padding-bottom: 0 }` and add bottom padding directly to `.tab` instead. The tab's accent border then sits flush against the divider with no offset.
- Option C — Cleanest: drop the `.tabHeader` border entirely and let the row of tabs supply the divider via a `box-shadow: 0 -2px 0 currentColor inset` or a tracking pseudo-element that spans full width.

Option B reads simplest and behaves consistently with most underline-tab designs.

### H4 — StringSetPicker inconsistency

Three callsites of the "shared" primitive produce three different label vocabularies:

| Callsite | Format | Example |
|---|---|---|
| ChordStringSetPicker (chord voicing) | Dot-separator from `strings: [0,1,2,3]` | `"1·2·3·4"` |
| FingeringPatternControls 1-string | Explicit `label: "String N"` | `"String 1"`, `"String 2"`, ... |
| FingeringPatternControls 2-strings | Dot-separator from `strings: [0,2]` | `"1·3"`, `"1·2"` |

The primitive itself is generic (`formatLabel` falls back to explicit label, else `allLabel` for `id==="all"`, else dot-format). The inconsistency comes from the CALLERS passing different option shapes.

The user's expectation: ONE consistent label vocabulary across all string-set selectors. Decisions to make:
- **Pick one format.** Options: (a) always `"String N"` / `"Strings N·M·..."`, (b) always pure dot format `"1·2·3·4"`, (c) abbreviation `"S1"`, `"S1·3"`. The dot format is most compact (good for narrow `fit` cells), but loses "guitar-string-ness" context. The word-prefixed variant is more discoverable.
- **Decide where strings-1-2-3 numbering anchors.** Guitar convention: string 1 = high E (treble). The current code uses `n+1` from 0-based indices already aligned to that convention.

Best UX call: **Use compact dot format everywhere** (`"1·2·3·4"`, `"3"` for a single string, `"All"` for the "all" id). With `fit` sizing, every callsite gets a tight trigger and the format scales from 1 string up to 6. A small uppercase prefix can disambiguate ("STRINGS 1·2·3·4") — but that prefix is what the `<Prop label>` already provides via the micro-label above. So inside the trigger, dot-only.

This drops the "String 1" wording from the 1-string select. The micro-label above the trigger reads "STRING" (single) or "STRINGS" (multi) per its `<Prop label>` — context preserved without redundancy.

### H5 — CAGED connectors persist after multi-shape select

**Reproduce:**
1. OVERLAY tab → SCALE: Pattern=CAGED, Shape=E (single shape).
2. CHORD: Voicing=Close. Connector polyline appears for the active chord's E-shape voicing.
3. Click another shape (Shift+click "D") so `cagedShapesAtom` now contains `{ E, D }`.

**Expected:** Either both shapes' connectors render together (if both are valid scale windows), or all connectors clear (with multi-shape, snap should disable like 1-string/2-string).

**Actual:** The original E-shape connector remains AND additional connectors stack on top. The user reads this as "ghost connectors not clearing."

**Cause (traced through atoms):**
- `activeScaleWindowAtom` (line 341 of `chordOverlayAtoms.ts`) — when `shapes.size > 1`, it returns `null` (no single window).
- `closeCandidatesAtom` — when `chordSnapToScaleAtom === true` AND `patternKeys.size > 0`, it filters. But when shapes > 1, `activeScalePatternPositionsAtom` returns the UNION of all active shapes' coordinates (per its caged branch via `cagedShapeDataAtom.highlightNotes` — and `cagedShapeDataAtom` in multi-shape mode aggregates ALL shapes' octaves). So `patternKeys.size > 0` still holds, and the filter still applies — but now allows voicings on ANY shape's coordinates.
- Result: many candidates pass, and each renders a connector. The user perceives the original (still-valid) connector as "stale" and the new ones as "additional ghosts."

**Plus the chord-overlay-connector hook (`useChordConnectorPolylines`) renders one polyline PER voicing match — with no de-duplication across shapes.**

**Fix design:**
- When `cagedShapes.size > 1`, treat Lock-to-scale as "no single window" and disable the filter (the same way it's disabled for 1-string/2-string). Auto-disable the toggle visually (`disabled` state).
- Optionally: stop rendering Close-voicing connectors when no single shape is active, OR show a clear single representative connector. Either way, the visual scatter must stop.

### H6 — Borrowed chord styling

**Current DegreeGrid CSS:**

```css
.cell.inKey {
  background: color-mix(in srgb, var(--faceplate-accent) 4%, transparent);
  /* default border via .cell */
}
.cell.borrowed {
  opacity: 0.55;
  border: 1px dashed color-mix(in srgb, var(--faceplate-divider) 50%, transparent);
  color: color-mix(in srgb, var(--text-muted) 60%, transparent);
}
.cell.selected {
  background: color-mix(in srgb, var(--faceplate-accent) 20%, transparent);
  border-color: var(--faceplate-accent);
}
```

**Problems:**
1. **`.borrowed.selected` has no dedicated rule.** When a user clicks a borrowed cell, both `.borrowed` (opacity 0.55) and `.selected` (20% accent bg) apply. The 0.55 opacity dims the 20% accent background to ~11% effective — selection barely reads.
2. **Numeral contrast:** Borrowed numerals (`♭ii`, `♯i`) drop to 60% of `--text-muted`, which on a dark faceplate is ~30-40% of body text color. Hard to read.
3. **Quality not surfaced.** Cells show only the note name + numeral. Since G11a defaults a borrowed root to "Major Triad", users have no in-grid signal of what chord they just picked.

**Design intent for the fix:**
- Selected borrowed cell: strong accent border + opaque background, no opacity dim. Reads as "selected first, borrowed second."
- Unselected borrowed cell: keep the dashed border + opacity-dim treatment, but raise opacity from 0.55 → ~0.7 so numerals stay legible.
- Add a small quality tag (e.g., `"M"`, `"m"`, `"°"`) under the note name on borrowed cells so the user sees what was assigned without leaving DegreeGrid.

## 3. Requirements

### H1. Fixed-width selects + layout overhaul (both tabs)

**Selects:**

- **Replace `fit` with `width` variants.** Add a per-instance width prop to `LabeledSelect` instead of (or alongside) `fit`. Three width modes:
  - `width="fill"` — current default (`100%` of cell). For long-label selects: Scale, Preset, Pattern.
  - `width="fixed"` — caller passes a fixed CSS length (`"6rem"`, `"8rem"`). Trigger uses that exact width regardless of content. Apply to: Root (`6rem`), Quality (`9rem`), Voicing (`6rem`), Lens-as-select if any, ChordStringSet (`8rem`), Scale 1-string/2-strings (`8rem`).
  - `width="auto"` — content-sized (today's `fit`). Reserved for selects whose content is predictable AND short.
- **Truncate overflowing trigger text** with ellipsis (`text-overflow: ellipsis; overflow: hidden; white-space: nowrap`).
- **Allow the dropdown menu to overflow the trigger.** Set `.labeled-select-content` `width: max-content` with a `min-width: var(--radix-select-trigger-width)`. Items can be wider than the trigger.

**PropGrid:**

- **Reduce `--prop-min-width`** from 11rem to ~8rem so cells pack tighter when their content is short. Cells with long content still grow via the `1fr` track.
- **Switch ChordOverlayControls' chord row from `columns={7}` (which doesn't math) to `columns={6}`** with spans Voicing 1 + Lens 2 + Strings 1 + Lock 2 = 6. Verify the actual rendered cell widths in the live UI.
- **Lens as a dropdown** (or stays as ToggleBar — designer's call). If it stays as ToggleBar, give it `flex: 0 0 max-content` so it sizes to content instead of stretching to fill span 2.

### H2. Lock-to-scale filter restored to fret-bound semantics

**Behavior:**

- When `chordSnapToScaleAtom === true` AND an applicable scale pattern is active, the filter applies.
- The filter now uses `activeScaleWindowAtom`'s `{ lo, hi }` fret bounds:
  ```ts
  windowed = all.filter((v) => {
    const fretted = v.positionKeys
      .map((k) => Number(k.split("-")[1]))
      .filter((f) => f > 0);
    if (fretted.length === 0) return true; // all-open voicing
    const min = Math.min(...fretted);
    const max = Math.max(...fretted);
    return min >= lo && max <= hi;
  });
  ```
- The position-key intersection filter (Plan G4) is **discarded**. It rejected too many useful voicings.
- `activeScalePatternPositionsAtom` (added in Plan G4) MAY stay for the disabled-options check in `stringSetOptionsAtom` (Plan G8), since that check benefits from precise position-key knowledge — but the `closeCandidatesAtom` filter no longer uses it.
- Rationale documented in code: chord tones that are NOT scale tones are still part of a valid voicing within the player's hand position. The lock-to-scale gate is about **hand position**, not **tonal subset**.

### H3. Tab underline aligns with divider

- `.tabHeader` `padding-bottom: 0`.
- `.tab` `padding: var(--space-2) var(--space-3) var(--space-2) var(--space-3)` — existing values, plus the bottom padding now provides the breathing room.
- The active tab's 2px accent border sits flush against `.tabHeader`'s 1px divider rail (visually one continuous line, with the accent dominating over the divider where they meet).

### H4. StringSetPicker uniform dot-only label format

- Drop `label: "String N"` from the 1-string callsite.
- 1-string: pass `options: [{ id: "0", strings: [0] }, ..., { id: "5", strings: [5] }]`. StringSetPicker formats as `"1"`, `"2"`, ..., `"6"`.
- 2-strings: keep current `strings: [a, b]` shape. Renders `"1·3"`, `"1·2"`, etc.
- Chord voicing: unchanged. Renders `"1·2·3·4"`, `"All"`.
- The `<Prop label>` above each trigger remains: "STRING" (singular) for 1-string, "STRINGS" (plural) for 2-strings + chord — provides the noun, the trigger carries only the numbers.
- Update tests to assert the new format.

### H5. Multi-shape CAGED auto-disables Lock-to-scale

- Extend the Lock-to-scale `disabled` gate in `ChordOverlayControls.tsx`:
  ```ts
  const lockToScaleApplies =
    (fingeringPattern === "caged" && cagedShapes.size === 1) ||
    fingeringPattern === "3nps";
  ```
- When disabled by this condition, the toggle stays visible (per H/G7 pattern) but greyed.
- Behavior: when shapes > 1, `chordSnapToScaleAtom` is functionally ignored (filter no-ops). No connector scatter from spurious matches.
- Verify the connector layer doesn't render stale polylines from a prior single-shape state when multi-shape mode activates. If it does, the connector hook needs explicit cleanup on `cagedShapes.size` transition — typically a simple recompute will suffice since the atom dependency chain will re-evaluate.

### H6. Borrowed-cell styling pass

- Add `.cell.borrowed.selected` rule: opacity 1, border `var(--faceplate-accent)`, background `color-mix(in srgb, var(--faceplate-accent) 20%, transparent)`. Selection dominates.
- Bump unselected `.borrowed` opacity from 0.55 → 0.7. Keep dashed border.
- Show a small quality tag under the note name on borrowed cells (e.g., `"M"`, `"m"`, `"°"`, `"M7"`). Requires DegreeGrid props to receive a quality-per-cell map OR a callback that resolves the borrowed quality at render time. The simpler path: compute the quality client-side in DegreeGrid via the same `guessQualityForBorrowedRoot` helper from G11a (exported from `progressionDomain.ts`).
