# FretFlow v2.0 Redesign — Design

**Status:** Brainstorm spec. Produced 2026-05-21. Single design covering three
loosely-independent subsystems that ship together as FretFlow Studio 2.0. After this
spec is approved, the work decomposes into three implementation plans (A / B / C) that
can be executed in order or in parallel.

**Date:** 2026-05-21 (revised 2026-05-21 after design review)

**Scope:** Collapse the Inspector from three tabs to two, retire outer cards, replace
the chord voicing system with a single Voicing control (Off / Full / Close) decoupled
from note highlights, gate Close voicing positions by hand-span physics and active
scale-shape window, and replace the chord root + degree controls with a unified
12-note degree grid with borrowed-note styling. Renames the Meter section, retires
Drop‑2, and migrates persisted state silently.

**Revision log:**

- 2026-05-21 v2: dropped String Set picker; merged Voicing + Connector-visibility
  into one Voicing dropdown with `Off` as a value; removed per-voicing Fret-range
  control (visibility is scale-driven at the View tab level); clarified Close
  positions cycle *within* the active scale-shape window (not across the whole neck);
  corrected Full voicing rendering rule (only the position matching the active scale
  shape, not all positions); moved Preset from Time to Progression; simplified smart
  quality default to "auto-fill on Add only"; dropped the `qualityIsManual` flag.

**Builds on:**

- `2026-05-19-voicing-controls-refinement-design.md` (engine fixes that this design
  obsoletes — drop2/triad cross-pollination becomes moot because drop2 is retired).
- `2026-05-19-controls-overhaul-design.md` (Inspector tab IA — v2.0 supersedes its
  four-tab layout with a two-tab one).

---

## 1. Background — why a v2.0

Live use of the current Inspector surfaced three categories of friction:

1. **Tab and card chrome cost more than it earns.** Three tabs (Scale / Chord / Song)
   with separate "card" wrappers around the Inspector and the progression Timeline make
   the page feel busier than the user's mental model. The Scale tab and the Chord tab's
   View controls overlap heavily; the user reads "fingering" and "voicing" as one
   coherent question — "what am I looking at on the neck?" — not two separate tabs.
2. **The voicing model is too dense and has known bugs.** Three voicing types
   (`caged` / `drop2` / `triad`), three inversions, four string-set windows, a
   `chordFretSpread` numeric stepper, and a `regionAtom` with four discrete values
   combine into a state space the user cannot reason about. The Drop‑2 / Triad engine
   has cross-pollination bugs (3-tone chords produce byte-identical Drop‑2 and Triad
   output — see `2026-05-19-voicing-controls-refinement-design.md` §1.1). The
   `chordFretSpread` is a fret-count, not a physical-width measure, so the same value
   is overly permissive low on the neck and overly restrictive high.
3. **Chord input requires two controls for one decision.** Picking a root requires
   either a Roman-numeral degree dropdown or a separate 12-note grid; the relationship
   between the two is opaque, and borrowed roots are not visually distinguishable from
   in-key roots.

v2.0 addresses all three in one coordinated release with a single design pass and
silent state migration.

## 2. Goals and Non-Goals

### Goals

- Reduce the Inspector to two tabs (**View**, **Song**) with no outer card chrome.
- Retire the **Drop‑2** voicing type entirely. Voicing types are **Full** and **Close**.
- Unify voicing-type and connector-visibility into one **Voicing** dropdown with
  values `Off` / `Full` / `Close`. `Off` removes all connector polygons.
- Decouple connector grouping from note highlighting — Lens picks highlights;
  Voicing picks connector grouping. They're independent.
- Replace the numeric `chordFretSpread` and `regionAtom` with a **hand-span physical
  filter** (Small / Medium / Large) and **scale-fingering-driven fret visibility** at
  the View-tab level — no per-voicing fret-range control.
- Display only **one Close voicing position at a time** inside the active scale-shape
  window. Prev/next cycles through 1–4 Close voicings inside that window; cycle
  resets when the scale shape changes.
- Full voicing displays the **single Full polygon matching the active scale shape**.
  No active fingering → all Full positions render simultaneously (existing behavior).
- Replace the Roman-numeral dropdown + 12-note grid with a single **12-note degree
  grid**, in which borrowed degrees are visually muted but selectable.
- Rename "Meter" → "Time" and surface the time signature in standard 4/4 form. Move
  the **Preset** picker from Time to Progression (it's a progression shape, not a
  tempo concern).
- Migrate stored state from v1 voicing types silently on first v2.0 launch.

### Non-Goals

- No changes to the Tonal.js theory layer, the scale catalog, or the audio engine.
- No new chord qualities, scales, or fingering patterns. The five fingering patterns
  (None / CAGED / 3NPS / 1-string / 2-string) stay.
- No restyle of the dark-mode color palette, semantic tokens, or motion language. Card
  removal is a structural change, not a visual one.
- No backing-track logic changes — the Song-tab Backing Track group stays exactly as
  in the mockup.
- No mobile-specific redesign — the existing responsive layout (mobile / tablet-split /
  tablet-stacked / desktop-split / desktop-stacked / desktop-3col) continues to gate on
  `data-layout-tier` / `data-layout-variant`; only the tab set and outer chrome change.
- No new lens types. Lens stays at "Chord tones" / "Lead" / etc.
- No "transition lines between voicings" feature — single binary connector visibility
  for v2.0; transition diagrams deferred to v2.1+.

## 3. Architecture overview

### 3.1 Two orthogonal layers

```
                                 ┌──────────────────┐
                                 │  HIGHLIGHTS      │
   Lens ────────────────────────▶│  (which notes    │
   (Chord tones / Lead / …)      │   get chord-tone │
                                 │   role styling)  │
                                 └──────────────────┘

                                 ┌──────────────────┐
                                 │  CONNECTORS      │
   Voicing ─────────────────────▶│  (Off / Full /   │
                                 │   Close polygon  │
                                 │   over notes)    │
                                 └──────────────────┘
```

- **Lens** controls which note instances get the chord-tone visual role on the
  fretboard. It does not draw connectors.
- **Voicing** controls the connector layer with three values: `Off` (no polygons),
  `Full` (full-chord polygon), `Close` (compact 3–5 string polygon). It does not
  filter highlights.
- The two layers are independent. Highlighting all chord tones while drawing a Close
  polygon over just three notes is a valid and useful state.

### 3.2 The "where on the neck" question has one driver

Both the **fret-visibility window** and the **active voicing's polygon position** are
governed by the same source of truth: the active **scale fingering pattern + position**.

- When a scale fingering is active (CAGED E shape, 3NPS pos 5, etc.), the fretboard
  visibly emphasizes the shape's fret window and any chord-tone notes outside that
  window are hidden (matching the existing scope-to-position behavior).
- The active Voicing polygon is the one that fits inside that same window:
  - **Full** voicing → exactly one Full polygon (the one for the active CAGED shape).
  - **Close** voicing → one of the 1–4 Close polygons inside the window, selected by
    a prev/next cycle.
- When no scale fingering is active, the fret window is the whole fretboard. Full
  voicings render at all 5 CAGED positions simultaneously; Close voicings render at
  all positions that pass the hand-span filter, one at a time via the cycle.

There is no per-voicing fret-range control — that's a job for the scale fingering
controls at the View-tab level, not a chord concern.

### 3.3 Hand-span physics

Connector position availability is computed from the physical width of a candidate
voicing in millimeters, using the standard 25.5" scale-length fret formula:

```
fret_position(n) = scale_length − scale_length / (2 ^ (n / 12))
```

A candidate Close voicing occupies frets `[f_low … f_high]`. Its physical width is
`fret_position(f_high) − fret_position(f_low − 1)`. If that width exceeds the user's
configured hand-span, the candidate is excluded from the cycle.

| Hand size | Threshold | Source                          |
| --------- | --------- | ------------------------------- |
| Small     | ~130 mm   | 5th–25th percentile adult       |
| Medium    | ~150 mm   | 50th percentile adult (default) |
| Large     | ~170 mm   | 75th–95th percentile adult      |

These are starting values, refinable in implementation. The control lives in the
existing **Settings overlay**, not in the View tab.

## 4. Subsystem A — Shell restructure

### 4.1 Tabs

The Inspector collapses from three tabs to two:

| Tab      | Contents                                                                          |
| -------- | --------------------------------------------------------------------------------- |
| **View** | Scale Fingering group + Chord Voicing group. Each with its own Show-on-Board toggle. |
| **Song** | Key group + Time group + Progression (master-detail) + Backing Track group.      |

Tabs render at the **top** of the Inspector panel as a text-tab list (Radix Tabs).
The numeric badge next to each tab name from the mockup ("View 1", "Song 2") is
**dropped** — the badges in the mockup are tab ordinals, not counters, and reading
them as counters is confusing. The tab IA stays bottom-docked under the fretboard for
all layout variants. The right-side-panel variant the user considered is **not**
adopted (research showed bottom docking is the right shape for a horizontally-dominant
fretboard).

### 4.2 Cards removed

The two outer "card" wrappers — the **Inspector card** and the **Progression Timeline
card** — are removed:

- Inspector content flows seamlessly to the bottom of the viewport against the page
  background. No border, no inset background, no shadow.
- The Timeline (top chord-pill strip: `1 Am · 2 Em · 3 G`) renders full-width against
  the page background, with the existing playhead behavior intact.
- The Status Bar runs full-width at the bottom edge.

**Internal section separators inside the View and Song tabs do not change.** The
existing "uppercase heading inline with description text" pattern stays. The user
explicitly clarified that "no cards" refers to the **outer** chrome only.

### 4.3 Inspector label removed

The accessible label `t("inspector.panelLabel")` and any visual "Inspector" heading
inside the Inspector panel are removed. The panel keeps a screen-reader-only
`<h2>Controls</h2>` or similar for landmark navigation; the visual heading is gone.

### 4.4 Status Bar full width

The Status Bar already renders mono left/right strips (Key, Chord, Lens, Pattern,
Frets on the left; Tempo, Tuning, "FretFlow Studio 2.0" on the right). It becomes
full-width by removing left and right insets that match the (now-removed) outer card
padding.

### 4.5 View tab content

```
┌─ SCALE FINGERING ──────────── How scale shapes are laid out on the neck.  [Show on Board ●]
│   Pattern: [CAGED system ▾]   Shape: [E shape ▾]   Position: [slider 0–24]
│
┌─ CHORD VOICING ────────────── How the active chord is voiced on the strings. [Show on Board ●]
│   Voicing: [Full ▾]           Lens: [Chord tones ▾]
│
│   When Voicing = Close:
│   Position: [◀ 2 / 4 ▶]
```

- **Voicing dropdown** — single control with three values:
  - `Off` — no connector polygons drawn. Chord-tone highlighting (via Lens) still
    applies. Equivalent to the v1 "Connectors = Hidden" state.
  - `Full` — one Full polygon at the active scale-shape position. If no scale shape
    active → all 5 Full positions render.
  - `Close` — one Close polygon at the active scale-shape window's cycle index. If no
    scale shape active → cycle spans all valid positions across the whole neck.
- **Position cycle** (only rendered when Voicing = Close): prev / index / total / next.
  Cycle membership = Close voicings that fit within the active scale-shape window AND
  pass the hand-span filter (§5.5). Cycle resets to index 0 when the scale shape
  changes. Keyboard ⬅ / ➡ advances the cycle when the Inspector has focus.
- **Lens dropdown** — Chord Tones / Lead / etc. Unchanged from v1, now placed in the
  Chord Voicing group rather than its own row.

There is **no separate String Set picker** (string set is implicit in the Close
position cycle), **no Fret range control** (the active scale fingering controls
visibility), and **no separate Connectors toggle** (replaced by `Voicing = Off`).

### 4.6 Song tab content

```
┌─ KEY ──────────────────────────────────────────────
│   Root note: [C][C#][D][D#][E][F][F#][G][G#][A][A#][B]
│   Scale: [Major ▾]
│
┌─ TIME ─────────────────────────────────────────────
│   Time signature: [4/4 ▾]   Tempo: [90 BPM]
│
┌─ PROGRESSION ────────────────── Preset: [Custom ▾]    [+ Add] [↑] [↓] [⧉] [🗑]
│   ┌─────────────────────────────┐  ┌───────────────────────────────────────┐
│   │ 1  [i]   A Minor Triad │1bar│  │ [i] A Minor Triad         CHORD 1 / 3 │
│   │ 2  [v]   E Minor Triad │1bar│  │                                       │
│   │ 3  [VII] G Major Triad │1bar│  │ ROOT — In-key degrees show their      │
│   └─────────────────────────────┘  │ Roman numeral; borrowed roots use     │
│                                    │ standard ♭/♯ notation.                │
│                                    │  [C ][C# ][D ][D# ][E ][F ][F# ]      │
│                                    │  [ i ][♭ii][ ii][♭iii][iii][iv][♯iv]  │
│                                    │  [G ][G# ][A ][A# ][B ]               │
│                                    │  [ v ][♭vi][vi ][♭vii][vii°]          │
│                                    │                                       │
│                                    │ QUALITY: [Minor ▾]                    │
│                                    │ DURATION: [− 1 +] [Beats | Bars]      │
│                                    │                                       │
│                                    │ Voicing & lens for this chord live    │
│                                    │ on the View tab.                      │
│                                    └───────────────────────────────────────┘
│
┌─ BACKING TRACK ────── Generated accompaniment that follows the progression.
│   Genre: [Rock ▾]      Lead Instrument: [Electric guitar ▾]
│   Chord Pattern: [Pop 8ths ▾]   Bass Pattern: [Root-Fifth ▾]
│   Drum Pattern: [Rock ▾]        Swing: [────●────] 0%
```

Notes:

- **Length** field is removed from the Time group. Total bars are derived from the
  step durations and shown only in the Status Bar (`bars · steps` summary already
  exists there).
- **Preset** moves from the Time group to the Progression group header. It's a
  progression shape (e.g., "I–V–vi–IV"), not a time-signature concern.
- **Master-detail layout**: progression list left, edit pane right; both rendered
  inline (no modal / popover state). On mobile / tablet-stacked, the list and edit
  pane stack vertically per the existing responsive variants.
- **Chord-tab cross-reference**: the hint "Voicing & lens for this chord live on the
  Chord tab" updates to "View tab" (Chord tab no longer exists).

## 5. Subsystem B — Chord voicing overhaul

### 5.1 Voicing types

```
                     v1                                  v2.0
                ────────────                       ────────────
                caged   ──────────────────────▶    full
                drop2   ──────────────────────▶    close (migrated)
                triad   ──────────────────────▶    close (migrated)
                                                   (drop2 retired)
```

- **Full**: a complete 5–6 string chord rendered as one polygon per CAGED-style
  position. Existing behavior of the old `caged` voicing type, renamed.
- **Close**: a 3–5 string chord rendered as one polygon over adjacent strings within a
  hand-span window. Replaces both `drop2` and `triad`. Includes triads (3-string
  voicings), tetrads (4-string seventh-chord voicings), and pentads (5-string ninth
  voicings) — chosen automatically from the chord's tone count.

The polygon shape for Close varies by tone count:

- 3-tone chord → 3-string polygon (triad).
- 4-tone chord → 4-string polygon (tetrad).
- 5-tone chord → 5-string polygon (pentad).

The user does not pick the tone count separately — it's determined by the chord's
quality (Major = 3-tone, Major 7 = 4-tone, Major 9 = 5-tone, etc.).

### 5.2 Connector grouping

Each voicing type defines its own polygon-generation rule. Both rules are pure
functions of `(chord, fretboardStringTuning, fretWindow)`:

- **Full**: for each CAGED position window across the fretboard (5 windows for a major
  triad in standard tuning), emit one closed polygon connecting all 6 strings'
  chord-tone vertices within that window. When a scale fingering is active, render
  only the polygon for the active CAGED shape. When no scale fingering is active,
  render all 5 polygons simultaneously.
- **Close**: for each adjacent string-set × position combination that contains all
  chord tones on 3–5 adjacent strings and passes the hand-span filter, emit one
  candidate polygon. When a scale fingering is active, the cycle is filtered to
  candidates whose fret range falls inside the active shape's window (plus a small
  buffer). When no scale fingering is active, the cycle spans all valid candidates
  across the neck. Exactly one polygon renders at a time — the one at the active
  Position cycle index. Total cycle length = `validCandidates.length`. When the
  active scale shape changes, the cycle resets to index 0.

### 5.3 Decoupled highlights

The **Lens** atom (`practiceLensAtom`) continues to govern highlight roles — Chord
Tones, Lead/Tension, etc. The voicing system does not filter highlights and does not
hide non-voicing chord tones. A user can highlight all chord tones across the
fretboard while the connector polygon outlines only a single Close position.

This is a behavioral change from v1, where `voicingMatchesAtom` filtered which chord-
tone instances received the chord-tone role. In v2.0, all chord-tone instances that
pass the Fret range filter get the role; the polygon is purely visual.

### 5.4 Multi-voicing display

The active scale fingering controls whether one or many voicing polygons render.

- **Full + active scale shape** → exactly one Full polygon (matching the shape).
- **Full + no scale shape** → all 5 Full polygons simultaneously (each is a different
  CAGED-position outline; no overlap because each is in a distinct fret window).
- **Close + active scale shape** → one Close polygon at a time. The polygon comes
  from the position cycle, filtered to candidates inside the shape's fret window.
- **Close + no scale shape** → one Close polygon at a time. The cycle spans all
  hand-span-passing candidates across the neck.

Overlap is impossible by construction: Full positions sit in non-overlapping CAGED
windows; Close only ever renders one polygon.

The Position cycle UI is the prev/next arrow pair from the mockup, with a counter
between them (`◀ 2 / 4 ▶`). Keyboard arrow keys ⬅ / ➡ advance the cycle when the
Inspector has focus.

### 5.5 Hand-span filter

The `handSizeAtom` (new) holds `"small" | "medium" | "large"`. It is persisted under
the existing storage prefix and surfaced **only in the Settings overlay**, not in the
View tab. Default = `"medium"`.

Hand-span is applied as a pure filter inside the voicing-engine output:

```ts
function filterByHandSpan(
  positions: ClosePosition[],
  handSize: HandSize,
): ClosePosition[] {
  const threshold = HAND_SPAN_THRESHOLDS_MM[handSize];
  return positions.filter((p) => physicalWidthMm(p) <= threshold);
}
```

`physicalWidthMm(p)` uses the 25.5" scale-length formula from §3.3. The filter
applies to Close only; Full positions are by definition within a CAGED window and
already span-bounded.

### 5.6 Drop‑2 retirement

The `drop2` voicing type, its engine path, its UI dropdown option, and its connector
palette slot are removed. The cross-pollination bug (3-tone chords producing
identical Drop‑2 and Triad output) becomes moot. The `2026-05-19-voicing-controls-
refinement-design.md` Goal #1 ("Triad and Drop 2 always produce visibly different
voicings") is superseded by drop‑2's removal.

Existing tests that assert Drop‑2 behavior are deleted; the test surface shifts to
Close.

## 6. Subsystem C — Chord input redesign

### 6.1 12-note degree grid

The current Roman-numeral dropdown + 12-note grid pair is replaced with a single
**12-note degree grid** in the Song-tab edit pane. Each cell shows two stacked
labels:

- **Note name** (top, larger): `C`, `C#`, `D`, `D#`, `E`, `F`, `F#`, `G`, `G#`, `A`,
  `A#`, `B`. Stored internally as sharps (existing convention); flats resolve at
  render via `getNoteDisplay(note, rootNote)`.
- **Degree numeral** (bottom, smaller):
  - In-key degrees use plain Roman numerals (`i`, `ii`, `iii`, `iv`, `v`, `vi`,
    `vii°`).
  - Borrowed degrees prefix with ♭ or ♯ (`♭ii`, `♭iii`, `♯iv`, `♭vi`, `♭vii`). The
    numeral is rendered in a **muted text color** (semantic token
    `--text-muted` or equivalent); the note name stays at full prominence.

The grid is one row on desktop (12 cells across), wrapping responsively on smaller
viewports.

### 6.2 Smart quality default — at step creation only

Smart defaults apply **only when a new progression step is created** (via the **+ Add**
button). On creation, the system picks a sensible Root and Quality based on the key:

| Step creation context | Default root             | Default quality                    |
| --------------------- | ------------------------ | ---------------------------------- |
| First step in empty progression | tonic (i / I)   | diatonic (minor for minor key, major for major key) |
| Subsequent steps      | next ascending in-key degree from the last step's root | diatonic for that degree |

Diatonic quality is resolved via the existing `getDiatonicChord()` wrapper in
`packages/core/src/lib/theory.ts:704-730`, which already covers all in-key cases
(Tonal's `Key.majorKey().triads` / `Key.minorKey().triads` underlying it).

**After creation, root and quality are independent inputs.** Changing the root via
the 12-note grid does **not** change quality, and vice versa. No `qualityIsManual`
tracking, no auto-overwrite. This matches user expectation: "I picked Cm — don't
turn it into C just because I clicked C."

Borrowed-chord defaults (♭III/♭VI/♭VII → Major) apply **only** when the user
explicitly chooses "Add borrowed chord" from the Add button's menu (a future
discoverable shortcut). For v2.0, picking a borrowed root from the 12-note grid does
not change quality from whatever the step already had. Custom borrowed-default code
is deferred — it's a v2.1 enhancement, not a v2.0 requirement.

### 6.3 Duration control

The Duration field stays as a `StepperControl` value with a `Beats | Bars` segmented
toggle. Same internal data model as today
(`{ value: number, unit: "beat" | "bar" }`). No behavior change.

### 6.4 Time section rename

The "Meter" section heading renames to **"Time"** (or "Time & Tempo" — final wording
during implementation review). The "Beats per Bar" field is replaced by a
**Time signature** picker showing standard `4/4` / `3/4` / `6/8` / etc. notation.
Implementation: a single combined dropdown sourcing from a `TIME_SIGNATURES` table;
the existing `beatsPerBarAtom` stays and a new `timeSignatureDenominatorAtom`
(default 4) is added.

The **Length** field is removed entirely. Total bars are derived from the sum of step
durations and shown in the Status Bar.

## 7. State migration

Persisted state is migrated silently on the user's first v2.0 launch, gated by a
bumped storage-schema version.

| v1 atom                       | v2.0 outcome                                  |
| ----------------------------- | --------------------------------------------- |
| `chordVoicingTypeAtom`        | Becomes 3-state `voicingAtom`: `"caged"` → `"full"`, `"drop2"` → `"close"`, `"triad"` → `"close"`. New value `"off"` if v1 had `voicingConnectorsAtom=false`. |
| `voicingInversionAtom`        | Retired. Position cycle starts at index 0. |
| `voicingStringSetAtom`        | Retired. String set is implicit in the Close position cycle. |
| `chordFretSpreadAtom`         | Retired. Replaced by hand-span filter (default Medium). |
| `regionAtom`                  | Retired. Fretboard visibility is scale-fingering-driven; no per-voicing override. |
| `voicingConnectorsAtom`       | Retired. Migrated into `voicingAtom` as `"off"` value. |
| `chordOverlayPracticeLensAtom`| Unchanged.                                    |
| `progressionStepsAtom`        | Unchanged. Existing steps keep their root + quality verbatim. New steps use the smart-default rule from §6.2. |
| (new) `handSizeAtom`          | Initialized to `"medium"`.                    |
| (new) `timeSignatureDenominatorAtom` | Initialized to `4`.                    |

A storage schema version key (`fretflow:storage-version`) tracks migration. v2 adds
`storageVersion: 2`. The migration runs once on read; failures fall back to defaults.

## 8. Decomposition into plans

This spec produces three implementation plans, executable independently:

### Plan A — Shell restructure

- Two-tab Inspector (View / Song).
- Remove outer Inspector card and Timeline card.
- Remove "Inspector" label.
- Full-width Status Bar.
- Remove Length field from Time group.
- Rename Meter → Time, add Time signature picker.

Touches: `Inspector/`, `MainLayoutWrapper/`, `ProgressionSummarySlot/`, `StatusBar/`,
`SongControls/`, `Inspector/tabs.tsx`.

Independent of B and C — ships its own value.

### Plan B — Chord voicing overhaul

- Retire Drop‑2 in engine + UI + tests.
- Rename `caged` voicing → `full`. Collapse the old voicing-type + connector-visibility
  atoms into a single 3-state `voicingAtom`: `"off" | "full" | "close"`.
- New `close` voicing path (3/4/5-string polygon generator on adjacent strings).
- Hand-span filter; `handSizeAtom` in Settings overlay.
- Position cycle UI (prev/next + counter) — only rendered when Voicing = Close.
  Cycle membership filtered by active scale-shape window when present.
- Decouple Lens from voicing-instance filter. Lens drives highlights only.
- Retire `regionAtom` and `chordFretSpreadAtom`. Fretboard visibility is scale-driven
  at the View-tab level — no per-voicing fret-range control.
- Drop String Set picker entirely (implicit in Close cycle).
- Full voicing renders only the polygon at the active scale shape (corrected from
  v1 spec wording that implied all positions always render).

Touches: `@fretflow/core` voicing engine, `chordOverlayAtoms.ts`,
`voicingCoupling.ts`, `ChordOverlayControls/`, `FretboardSVG/hooks/useChord*`,
`SettingsOverlay/`.

Depends on Plan A only for the renamed Voicing dropdown placement in the View tab —
otherwise independent. Could ship behind a feature flag if A is not ready.

### Plan C — Chord input redesign

- 12-note degree grid component in Song-tab edit pane (replaces separate degree
  dropdown + root grid pair).
- Borrowed-degree styling (muted numeral, full-prominence note name).
- Smart default at step *creation* only: new "+ Add" step picks next ascending in-key
  degree + its diatonic quality via `getDiatonicChord()`. No `qualityIsManual` flag;
  root and quality are independent after creation.
- Update "live on the Chord tab" hint → "View tab" (depends on A's tab rename
  landing).

Touches: `SongControls/`, `shared/DegreeSelect.tsx` (deprecated),
`shared/RootNoteSelect.tsx` (extended), `shared/ChordQualitySelect.tsx`,
`progressionAtoms.ts`.

Depends on Plan A for the View-tab reference text. Otherwise independent.

### Recommended order

1. **Plan A first** (shell). Lowest-risk visual/structural work; unlocks the V2 page
   layout and removes friction sources first.
2. **Plan B second** (voicing). Highest engineering depth; benefits from Plan A's tab
   IA being settled.
3. **Plan C third** (chord input). Smallest scope; lands the final visible polish.

Plans B and C can be executed in parallel by separate sessions once Plan A merges.

### Implementation skill: frontend-design

All three plans' implementation phases use the **`frontend-design`** skill for the
visual and component-design work — the two-tab Inspector layout (A), the View-tab
Voicing controls including the Position cycle, the Settings overlay Hand Size picker
(B), and the 12-note degree grid with borrowed-degree styling (C). Each plan's task
list cites frontend-design at the relevant component-build steps so the executing
session loads the skill before touching the UI.

The existing FretFlow visual language (tokens, semantic CSS, motion patterns) stays —
frontend-design is used for *layout, hierarchy, and interaction polish* on the new
controls, not for a wholesale visual restyle.

## 9. Non-decisions / explicitly deferred

The following came up during brainstorming and are **deliberately deferred** past v2.0:

- **Transition-between-voicings lines** ("Show between voicings" mockup wording). Not
  shipped. Single binary Visible/Hidden toggle for v2.0.
- **Manual position lock / custom fret-range zoom**. Beyond the Auto / Whole binary
  for v2.0. Future fret-range work goes through its own brainstorm cycle.
- **Right-side-panel layout variant** for the View tab on wide desktops. The bottom
  tab placement is the v2.0 layout for all variants.
- **Per-step backing-track override** (e.g., one chord uses a different chord-pattern
  than the rest). Backing Track stays a progression-level set of settings.
- **Lefty / alternate-tuning hand-span recalibration**. v2.0 hand-span uses standard
  25.5" scale length and standard-tuning fret math. Multi-scale (fanned) tunings and
  short-scale instruments are not modelled.
- **CAGED-shape carousel for Full voicing** (one-at-a-time cycle through E, D, C, A, G
  shapes). v2.0 renders all Full positions simultaneously, as today.

## 10. Open questions / implementation-time choices

These are deliberately left for the plan and implementation phases — they don't block
the design:

- Exact Time-section heading copy: "Time" vs "Time & Tempo" vs "Tempo & Time
  signature." Decided during plan review with UI copy in hand.
- Exact `HAND_SPAN_THRESHOLDS_MM` values for Small / Medium / Large. Starting at
  130 / 150 / 170 mm; tuneable after live trial.
- Whether the 12-note degree grid wraps to two rows on narrow widths or stays on one
  with smaller cell sizes. Falls out of the existing responsive breakpoints during
  implementation.
- Buffer size when filtering Close cycle candidates to the active scale-shape window
  (the active CAGED E shape covers frets 0–4; do we include candidates whose notes
  spill 1 fret outside? 2 frets?). Default starting value: 1 fret of buffer either
  side. Tuneable.

## 11. Success criteria

v2.0 ships when:

1. The Inspector renders two tabs (View / Song), no outer Inspector card, no outer
   Timeline card, full-width Status Bar, no visible "Inspector" label.
2. Voicing dropdown shows exactly `Off`, `Full`, and `Close`. No `Drop‑2` option,
   no `Triad` option, no separate connector-visibility toggle, no String Set picker,
   no per-voicing Fret-range control.
3. A user with stored v1 voicing of `drop2` or `triad` lands on Close after first
   launch with no error and no migration UI. A user with v1 `voicingConnectors = false`
   lands on `Voicing = Off` after migration.
4. Selecting Close with an active scale shape shows a Position cycle whose
   candidates all fall inside the scale-shape window (+ buffer) and all pass the
   hand-span filter. Rendering at any cycle index draws exactly one polygon.
5. Selecting `Hand size = Small` reduces the Close Position cycle length (or leaves
   it unchanged where no positions exceed Small's threshold). Selecting `Large`
   never decreases the cycle length compared to `Small`.
6. Lens = Chord Tones with Voicing = Close highlights all chord-tone instances on the
   fretboard (subject to the scale-fingering visibility window), not just those
   inside the active Close polygon.
7. Voicing = Full with an active scale shape renders exactly one Full polygon (the
   one matching the shape). Voicing = Full with no scale shape renders all 5 Full
   polygons.
8. The Song-tab edit pane shows a 12-note degree grid with borrowed degrees visibly
   muted, and clicking + Add for a new step picks the next in-key degree with its
   diatonic quality. Subsequent root or quality edits on that step do not auto-change
   the other field.
9. The Time section shows `Time signature: 4/4` (not `Beats per Bar: 4`), no Length
   field, and no Preset control (Preset has moved to the Progression group header).
10. The Status Bar's `bars · steps` summary updates as the user adds / removes
    progression steps (replaces the removed Length field).
11. `pnpm run lint && pnpm run test && pnpm run build` pass on all three plans'
    branches.
